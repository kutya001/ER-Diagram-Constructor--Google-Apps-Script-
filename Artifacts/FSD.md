## 1. Обзор системы

### 1.1 Назначение

ER Diagram Constructor — визуальный конструктор Entity-Relationship диаграмм, реализованный как веб-приложение Google Apps Script. Позволяет проектировать схемы баз данных с таблицами, колонками и связями на интерактивном канвасе. Все данные хранятся в Google Таблицах.

### 1.2 Целевая аудитория

- Разработчики и проектировщики баз данных
- Аналитики, создающие прототипы схем данных
- Команды, использующие Google Workspace

### 1.3 Ключевые возможности

| Возможность | Описание |
|-------------|----------|
| Управление схемами | Создание, копирование, удаление схем БД |
| Интерактивный канвас | Drag & drop таблиц, zoom/pan, мини-карта |
| Авторасположение | 5 алгоритмов компоновки таблиц |
| Таблицы и колонки | Полный CRUD с типизацией (13 типов данных) |
| Визуальные связи | FK-связи рисуются на канвасе перетаскиванием |
| Шаблоны таблиц | Переиспользуемые шаблоны с предустановленными колонками |
| Справочники | Управление категориями, назначениями, типами колонок |
| JSON экспорт/импорт | Экспорт и импорт схем, генерация AI-промптов |
| Экспорт в Google Таблицу | Создание отдельной таблицы из схемы |
| Sheet Builder | Управление листами Google Таблицы |
| Темизация | 3 цветовые темы (тёмная, светлая, тёплая) |

## 2. Архитектура

### 2.1 Стек технологий

| Слой | Технология |
|------|------------|
| Среда выполнения | Google Apps Script (V8) |
| Бэкенд | `.gs` файлы — серверный JavaScript |
| Фронтенд | Vanilla JS + Canvas API в `.html` файлах |
| Хранилище | Google Sheets (9 листов) |
| Деплой | GAS Web App через `doGet()` |
| Разработка | clasp (CLI) |

### 2.2 Коммуникация клиент–сервер

Вся коммуникация происходит через `google.script.run` (HTML Service API). Вызовы асинхронны и обёрнуты в Promise.

| Функция | Серверный метод | Назначение |
|---------|----------------|------------|
| `api(action, params)` | `processRequest(action, params)` | Одиночный запрос |
| `apiBatch(reqs)` | `processBatch([{action, payload}])` | Пакетная отправка |
| `apiAll()` | `loadAllData()` | Начальная загрузка всех данных |

Ответ всегда имеет формат: `{ ok: true, data: <result> }` или `{ ok: false, error: <message> }`.

### 2.3 Синхронизация (Sync Engine)

Мутации помещаются в очередь `SQ` через `queueSync(action, payload)` и отправляются с debounce-задержкой 400 мс одним пакетным запросом `processBatch`.

**Состояния синхронизации:**

| Состояние | Индикация |
|-----------|-----------|
| `pending` | Оранжевые точки, полоса 25% |
| `syncing` | Вращающийся значок, полоса до 90% |
| `synced` | Зелёная галочка, полоса 100% → скрытие |
| `error` | Красный крест, до 3 повторов через 2 с |

При исчерпании попыток несохранённые действия сохраняются в `localStorage('er_failed_sync')` для ручного повтора.

### 2.4 Оптимистичный UI

При создании и копировании таблиц данные немедленно добавляются в состояние с временным ID (`tmp_`-префикс), канвас перерисовывается, затем фоновый API-запрос заменяет временные данные на реальные серверные ID.

- Иерархическое дерево: корневые схемы (📁) и копии (↳)
- Каждый узел: название, badge с количеством таблиц, кнопка «↩ Отвязать» для копий
- Клик — выбор схемы; правый клик — контекстное меню

**Правая часть — Список таблиц:**
- Таблицы сгруппированы по категориям (сворачиваемые группы)
- Каждая строка: точка статуса GSheet (зелёная/серая), название, кол-во колонок
- Клик — выбор таблицы с центрированием на канвасе

**Настройки размеров:** сохраняются в localStorage (`er_sb_w`, `er_sbl_w`). Каждая часть независимо сворачивается.

### 5.4 Канвас

Интерактивная область рисования на основе Canvas API.

**Модель координат:**
- Мировые координаты ↔ экранные: `w2s(wx,wy)`, `s2w(sx,sy)` с учётом смещения `ox,oy` и масштаба `sc`
- Масштаб: 0.2× – 3×

**Отрисовка (RAF-управляемая):**
1. Фон + сетка (40px шаг)
2. Кривые Безье для FK-связей (с стрелками)
3. Карточки таблиц (заголовок, категория, колонки, FK-маркеры)
4. Черновик создаваемой связи (пунктирная линия)

**Взаимодействие:**

| Ввод | Действие |
|------|----------|
| Клик по таблице | Выбор таблицы |
| Перетаскивание таблицы | Перемещение на канвасе |
| Перетаскивание пустой области | Панорамирование |
| Колесо мыши | Зум с центром в точке курсора |
| Двойной клик по таблице | Открыть редактирование |
| Правый клик по таблице | Контекстное меню |
| Перетаскивание от FK-точки к PK | Создание связи |
| Клик по кривой связи | Всплывающее окно с деталями связи |
| Наведение на FK-точку | Курсор → crosshair |
| Наведение на кривую связи | Курсор → pointer, подсветка линии |

**Хит-тесты:**
- `hitTable(wp)` — точка в прямоугольнике таблицы
- `hitFK(wp)` — радиус 8px от FK-маркера
- `hitPK(wp, excludeId)` — аналогично для PK
- `hitRel(wp)` — сэмплирование кривой Безье с шагом 5%, порог 8/scale px

**Мини-карта:** уменьшенный обзор всех таблиц с рамкой видимой области (150×95px, правый нижний угол).

## 6. Пользовательские сценарии

### 6.1 Создание схемы и таблицы

1. Пользователь нажимает «＋» рядом с селектором схем
2. Открывается модалка создания схемы
3. Вводит название и описание → «Сохранить»
4. Схема появляется в дереве и селекторе, автоматически выбирается
5. Нажимает «＋ Таблица» → открывается модалка таблицы
6. Заполняет: название, категория (автозагрузка колонок шаблона), назначение, описание
7. Добавляет колонки через «＋ Столбец», настраивает типы, перетаскивает для порядка
8. «💾 Сохранить» → таблица мгновенно появляется на канвасе (оптимистичный UI)

### 6.2 Создание связи на канвасе

1. Наведение на FK-маркер таблицы → курсор становится crosshair
2. Нажатие (mousedown) начинает режим рисования связи
3. Пунктирная линия следует за курсором
4. Отпускание (mouseup) на PK-колонке другой таблицы → связь создаётся
5. Toast: «Связь создана ✓»

### 6.3 Создание связи через модалку

1. Нажимает «🔗 Связи» → открывается модалка
2. В форме «＋ Добавить связь»: выбирает FK-таблицу → FK-колонку → PK-таблицу → PK-колонку
3. «Создать» → связь добавляется в список

### 6.4 Генерация схемы через AI

1. Открытие «{ } JSON» → вкладка «🤖 ИИ Промт»
2. Копирование сгенерированного промпта с системными данными
3. Вставка в ChatGPT/Claude/Gemini с описанием нужной схемы
4. Получение JSON от AI
5. Переключение на вкладку «📥 Импорт JSON»
6. Вставка JSON → «✓ Проверить JSON» → «📥 Импортировать схему»
7. Схема загружается и автоматически выбирается

### 6.5 Экспорт в Google Таблицу

1. Нажатие «📤 В таблицу» → модалка экспорта
2. Опционально: название книги, папка в Drive, примеры данных
3. Превью показывает список листов, которые будут созданы
4. «📤 Создать таблицу» → прогресс → ссылка на созданную таблицу

### 6.6 Sheet Builder

1. Нажатие «🏗 Листы» → модалка Builder
2. Вкладка «Управление»: статус каждой таблицы (лист существует / отсутствует)
3. Пакетные операции: создать недостающие, пересоздать все, удалить все
4. Индивидуальные операции через чекбоксы

## 7. Бизнес-правила

### 7.1 Копирование схемы

- Создаётся новая схема с суффиксом ` (копия)` и `copied_from` = ID оригинала
- Глубокое копирование всех таблиц и колонок
- FK-связи **внутри** копии перепривязываются к новым ID (внутренняя консистентность)
- FK-связи на таблицы **вне** схемы очищаются (не переносятся)
- Позиция: смещение +20px по X и Y

### 7.2 Каскадное удаление

| Удаление | Каскад |
|----------|--------|
| Схема | Все колонки таблиц схемы → все таблицы схемы → схема |
| Таблица | Все колонки таблицы → таблица |
| Шаблон | Все колонки шаблона → шаблон |

### 7.3 Защита ссылочной целостности при удалении

| Сущность | Проверка |
|----------|----------|
| Тип колонки | Проверка использования в COLUMNS и TEMPLATE_COLS. Блокировка при наличии |
| Категория | Проверка использования в TABLES. Блокировка при наличии |
| Назначение | Проверка использования в TABLES. Блокировка при наличии |

### 7.4 Применение шаблона

- Только колонки, **не существующие по имени** (регистронезависимо) в таблице, добавляются
- Возвращает массив фактически добавленных колонок

### 7.5 Создание таблицы с шаблоном

- При `apply_templates=true` и заданном `category_id` — автоматически применяются все шаблоны указанной категории + все универсальные шаблоны (с пустым `category_id`)

### 7.6 Импорт JSON

- **Обычный режим:** ошибка, если схема с таким именем уже существует
- **Режим перезаписи** (`overwrite: true`): удаление старых таблиц/колонок, запись новых
- Автосоздание отсутствующих типов колонок, категорий, назначений (сопоставление по имени)
- FK-ссылки разрешаются в памяти (3-проходный алгоритм: таблицы → колонки → FK-патч)
- Автопозиционирование таблиц: `sqrt(N)` колонок, 280×220px шаг

### 7.7 Копирование таблицы

- Суффикс ` (копия)`, смещение +30px
- Все колонки копируются индивидуально

### 7.8 Автоинкремент позиции

При создании колонки / колонки шаблона без указания `position` — автоматически вычисляется `maxPosition + 1`.

### 7.9 Аудит дат

- `create_date_time` — устанавливается один раз при создании
- `update_date_time` — обновляется автоматически при каждом `_updateRow`

## 8. Валидация данных

| Контекст | Правило |
|----------|---------|
| FK-цель при создании/обновлении колонки | Если `is_fk=true`: целевая таблица должна существовать; целевая колонка (если указана) должна существовать |
| Тип колонки — имя и обозначение | Обязательны (непустые) при создании и обновлении |
| Тип колонки — уникальность обозначения | Дубликат designation блокируется |
| Категория / Назначение — имя | Обязательно (непустая строка) |
| JSON импорт — формат | Обязательны поля `schema.name` и `tables[]` |
| JSON импорт — дубликат схемы | Ошибка, если имя уже занято (без `overwrite`) |
| Boolean-значения | `isTrue(v)`: принимает `true`, `'true'`, `'TRUE'` (особенность Google Sheets) |

## 9. Обработка ошибок

### 9.1 Серверная сторона

| Уровень | Поведение |
|---------|-----------|
| `processRequest` | Каждое действие обёрнуто в try/catch → `{ok: false, error: message}` |
| `processBatch` | Каждый запрос изолирован; один сбой не прерывает остальные |
| Валидация | `throw new Error(message)` с описательными русскоязычными сообщениями |
| `_updateRow` | Возвращает `null` при отсутствии строки (тихий сбой) |
| `_deleteRow` | Возвращает `false` при отсутствии строки |

### 9.2 Сообщения об ошибках

| Сообщение | Контекст |
|-----------|----------|
| «Целевая таблица FK не найдена: X» | Валидация FK при создании/обновлении колонки |
| «Обозначение уже используется: X» | Дубликат designation типа колонки |
| «Название и обозначение обязательны» | Пустые поля типа колонки |
| «Название обязательно» | Пустое имя категории / назначения |
| «Категория используется в таблицах — удаление невозможно» | Защита от удаления связанной категории |
| «Назначение используется в таблицах — удаление невозможно» | Защита от удаления связанного назначения |
| «Тип используется в колонках таблиц — удаление невозможно» | Защита от удаления используемого типа |
| «Тип используется в шаблонах — удаление невозможно» | Аналогично для шаблонов |
| «Schema not found» / «Table not found» | Операции копирования |
| «Схема ... уже существует» | Импорт JSON без overwrite |

### 9.3 Клиентская сторона

- Ошибки API отображаются через toast-уведомления типа `err`
- Sync Engine сохраняет несинхронизированные данные в `localStorage` после 3 неудачных попыток
- Ручной повтор через `retryFailedSync()`
